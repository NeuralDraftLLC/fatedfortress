## What is Fated Fortress?

Fated Fortress is a structured task marketplace built for AI-scoped digital work — 3D assets, audio, code, design. Hosts post a project brief, an AI engine decomposes it into precise tasks with machine-readable specs and payouts. Contributors claim tasks knowing exactly what the deliverable must be. When they submit, an automated deep-spec gate validates the asset before a human ever reviews it.

Payment is pre-authorized at claim time and captured only on approval. No funds in escrow — no risk on either side.

---

## The Core Loop

```
Host posts a project brief
  └─ AI (GPT-4o) decomposes the brief into tasks
       Each task carries: payout range · deliverable type · spec_constraints
         └─ Contributor browses marketplace → claims a task
              Stripe PaymentIntent created at claim time (manual capture)
                └─ Contributor delivers the asset
                     └─ verify-submission parses binary headers
                          ├─ Spec mismatch? Auto-rejected. Contributor is notified.
                          └─ Spec passes? Enters host review queue.
                               └─ Host approves → Stripe captures → payout released
                                  Host ignores 48h → auto-approved, contributor paid
```

## Why the Architecture Is Different

### Payment Protected Before Work Begins

Most gig platforms authorize payment at approval. Fated Fortress creates a Stripe `PaymentIntent` with `capture_method: manual` the moment a task is claimed. The host’s payment method is validated before a contributor types a single line or renders a single polygon. If the host has no valid funding, the claim fails — immediately, not after days of wasted work.

### Machine-Readable Briefs

Every task carries a `spec_constraints` JSON field generated by the Asset Scanner. It is not a prose description — it is a contract. For a 3D model task, that means `{ max_polygons: 100_000, requires_rig: true, lod_levels: 2 }`. The verification gate reads that field at submission time, parses binary file headers, and enforces the spec with zero human judgment required.

### Isolated AI Keystore

API keys never touch the main application thread. The keystore runs in a sandboxed worker at an isolated origin. The browser’s same-origin policy is the enforcement boundary — not a promise, not an env check, the OS-level security model.

### Real-Time Collaborative Review

The host review screen runs a live Y.js CRDT over WebRTC, relayed through Cloudflare Durable Objects. Multiple reviewers see each other’s edits, cursors, and presence in real time. State is automatically reconciled — no WebSocket server to manage, no polling.

## System Architecture

Three independently deployable applications. No monorepo tooling required.

| App | Role | Stack |
|-----|------|-------|
| `apps/web` | User-facing SPA | TypeScript · Vite |
| `apps/worker` | Isolated AI keystore + provider adapters | Vite IIFE · Web Workers |
| `apps/relay` | Y.js signaling · WebRTC · TURN credentials | Cloudflare Workers · Durable Objects |

### Full Stack

| Layer | Technology |
|-------|------------|
| Database + Auth | Supabase — Postgres, Row-Level Security, Edge Functions |
| Payments | Stripe Connect Express — manual capture at claim time, 10% platform fee |
| Real-Time / CRDT | Y.js over WebRTC · Cloudflare Durable Objects relay |
| AI Scoping | OpenAI GPT-4o via `scope-tasks` Edge Function |
| Asset Intelligence | 9-sub-pass layered scanner (deterministic → heuristic → gap analysis) |
| Deep-Spec Verification | Binary header parsing — GLB, WAV, MP3, PNG, JPEG |
| Object Storage | Supabase Storage with presigned PUT URLs |
| Hosting | Cloudflare Pages (web) · Cloudflare Workers (relay) |
| Error Tracking | Sentry — PII scrubbed via `packages/sentry-utils` |

## Data Model

```
Project
  └─ Task
       payout_range · deliverable_type · spec_constraints · status · version
         └─ Submission
              deliverable_url · proxy_video_url · revision_count
                └─ Decision
                     verdict (approve/reject/revision) — authoritative for payout
```

**Project Wallet** — per project, three buckets: `deposited`, `locked`, `released`. Every mutation goes through an atomic Postgres RPC with `SELECT FOR UPDATE`. No client-side arithmetic. No race conditions.

### spec_constraints Examples

| Deliverable Type | Spec Shape |
|-----------------|------------|
| `3d_model` | `{ max_polygons: 100000, requires_rig: true, lod_levels: 2 }` |
| `audio` | `{ sample_rate: 44100, channels: 2, bit_depth: 16 }` |
| `design_asset` | `{ max_width: 2048, max_height: 2048, min_width: 256 }` |

## Edge Functions

| Function | Trigger | What it does |
|----------|---------|--------------|
| `scope-tasks` | On demand | GPT-4o decomposes host brief → Zod-validated tasks → persisted as `draft` |
| `asset-scanner` | On demand | 9-sub-pass analysis: magic-byte detection → heuristic inference → gap bounty calc |
| `asset-sanitizer` | Submit | ClamAV scan + EXIF/metadata strip + GLB proxy turntable render — **new** |
| `create-payment-intent` | Claim | Version-checked atomic claim; creates Stripe PI only on version match |
| `verify-submission` | Submit | Parses binary headers; enforces `spec_constraints`; auto-rejects on mismatch |
| `stripe-webhook` | Stripe event | Handles `payment_intent.succeeded/failed`, `transfer.created`, `account.updated` |
| `stripe-payment` | On demand | PI capture / cancel / refund / transfer |
| `auto-release` | Cron 30 min | 24h warning → 48h auto-approve + capture + wallet release |
| `expire-claims` | Cron 5 min | Reclaims expired soft-locks; returns funds via `unlock_wallet` RPC |
| `stripe-connect-onboard` | On demand | Stripe Connect Express onboarding flow |
| `github-oauth` | OAuth callback | Server-side GitHub token exchange |
| `supabase-storage-upload` | On demand | Generates presigned PUT URLs for Supabase Storage |

## Getting Started

### Prerequisites

- Node.js 18+
- A Supabase project (apply migrations 001–009)
- Stripe account with Connect enabled
- Cloudflare account (for relay deployment)

### Local Development

```bash
git clone https://github.com/NeuralDraftLLC/fatedfortress.git
cd fatedfortress

# Web app
cd apps/web
npm install
cp .env.example .env.local
# Set VITE_SUPABASE_URL and VITE_SUPABASE_ANON_KEY
npm run dev
# → http://localhost:5173

# Isolated AI keystore (optional, separate origin)
cd apps/worker
npm install
npx vite --port 5174

# Y.js relay (optional)
cd apps/relay
npm install
npx wrangler dev
```

**Full environment reference:** [`env-vars.md`](env-vars.md)

### Required Edge Function Secrets

```
STRIPE_SECRET_KEY
STRIPE_WEBHOOK_SECRET
OPENAI_API_KEY
GITHUB_TOKEN
CLAMAV_API_KEY          # Pillar 3 — antivirus scan service
```

### E2E Tests

Full-path smoke test: forge → claim → upload → sanitize → verify-submission → approve → Stripe capture.

```bash
npm install                     # repo root
npx playwright install chromium
cp e2e/.env.example e2e/.env
# Fill in dev Supabase + Stripe credentials
npm run test:e2e
```

Requires a real Supabase dev project. See [`e2e/README.md`](e2e/README.md).

## Contributing

**Adding a route:** Register in `apps/web/src/main.ts` and create the entrypoint in `apps/web/src/pages/`. Router registration automatically handles Y.js room teardown — no manual cleanup needed.

**Adding shared types:** `packages/protocol/src/index.ts`.

**New Edge Functions:** Add under `supabase/functions/`; use `_shared/auth.ts` for auth. Session JWTs for user-facing invokes, `CRON_SECRET` for cron endpoints.

**New migrations:** Date-prefixed files (`20260501_XXX.sql`). Test locally with `supabase db push`. Columns must exist before RPCs that reference them.

**New deliverable types:** Add spec shape to `buildSpecConstraints()` in `asset-scanner/index.ts`, add binary checker to `verify-submission/index.ts`, and add re-encode handler to `asset-sanitizer/index.ts`.

**CSP / new origins:** Update `connect-src` in `apps/web/index.html` when adding new API endpoints or external origins.
